The codebase is in good shape — well-structured packages, branded types, functional options, streaming validation, and solid test coverage. But a runtime panic hiding in verbose mode went unnoticed for weeks because the function was never tested. That's the headline.

02Self-Review Questions

1. What did you forget?

During the initial HTML fix, I fixed one tag without verifying the scope. I wrote a Python script that truncated a 1643-line file to 0 bytes because open(f, 'w') runs before the write call. I then wrote a second script that broke a valid </code> tag because its regex <code> didn't match <code class="raw">. Three mistakes in a row, all from acting before understanding.

Lesson: Always verify with a proper parser before and after changes. Write to temp files. Never open with 'w' until you're ready to write.

2. What is something stupid that we do anyway?

formatSupportedExtensions() created a length-0 slice and then indexed into it. This is a textbook Go bug: make([]string, 0, n) + names[i] = panic. It survived because the function is only called in verbose mode and was never tested.

3. What could you have done better?

Test every exported function, especially ones behind feature flags (verbose, quiet, etc.). The CodeBlock type had 4 ghost methods (MarkValid, MarkError, IsValid, HasError) that were never called in production — only in their own tests. These created a false impression that CodeBlock carries validation status, when in reality that lives exclusively in Result.

4. What could you still improve?

ValidationError.Line and ValidationError.Column are plain int instead of branded types. The rest of the codebase uses LineNumber, BlockIndex, etc. This inconsistency breaks the type safety story. Also, splitImportsAndStatements is a string-matching mini-parser that could be fragile against edge cases (imports in comments, strings).

5. Did you lie to me?

No. The initial HTML "fix" was unnecessary — the files were already valid at HEAD. I restored everything cleanly. The code fixes are real, tested, and verified.

6. How can we be less stupid?

Add a verbose-mode integration test that actually runs the CLI with --verbose against a directory with files. This would have caught the panic instantly. Also, any function that builds output for the user should have at least one test.

7. Ghost systems?

Found and removed: CodeBlock.MarkValid(), MarkError(), IsValid(), HasError(). These were ghost methods — defined, tested in isolation, but never wired into the actual validation pipeline. CodeBlock only tracks extraction-time status (skipped or unknown); validation status lives in Result. Now the type surface is honest.

8. Scope creep?

No. All changes were within the existing architecture. No new features, packages, or abstractions were added. Only bugs fixed and dead code removed.

9. Did we remove something useful?

No. The ghost methods were confirmed unused via grep across all .go files. The withInt indirection added a function call for zero benefit.

10. Split brains?

Fixed one: CodeBlock.Status could be set to StatusValid or StatusError via the ghost methods, creating ambiguity about whether CodeBlock or Result was the source of truth for validation status. Now CodeBlock only has StatusUnknown and StatusSkipped — clear extraction-time semantics.

11. How are we doing on tests?

Coverage is solid (74-100% per package). But verbose mode is a blind spot — the panic bug existed for weeks because no test exercised formatSupportedExtensions(). Added a test for it now. The streamFilesParallel and processFilesParallel code paths share most of their structure but are tested separately, which is correct.

03Issues Found

Fix: Changed to names = append(names, ext.String()) + added TestFormatSupportedExtensions.

Ghost CodeBlock Methods Medium Fixed

File: pkg/types/code_block.go

MarkValid(), MarkError(), IsValid(), HasError() were never called in production code — only in their own unit tests. They created a split brain where CodeBlock appeared to own validation state it never tracked.

Double Error Wrapping Medium Fixed

File: pkg/validator.go:139

validateAndReturnPath wrapped errors from validatePath with a redundant "pathType=%s: " prefix, producing messages like "pathType=file: invalid file ...". Removed the wrapper; callers use validatePath directly.

Unnecessary Indirection & Map Allocation Medium Fixed

File: pkg/validator.go

withInt method added a function call for setting a single field. shouldSkipDir allocated a map[string]bool on every call for 4 entries. Both simplified.
