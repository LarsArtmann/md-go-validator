Making md-go-validator superb to embed

A review of md-go-validator as a Go library that other tools (BuildFlow, CI runners, editors, doc-site generators) would import and call in-process — the dimension the sibling CLI/accuracy review does not cover. The core engine is excellent. The contract around it is what decides whether anyone embeds it.

00The verdict

As an engine — the extractor, the 5-strategy Go parser, the tree-sitter validators, the invariant-enforcing Result type — md-go-validator is genuinely good. As a library that another Go program imports and drives in-process, it is one decision away from being embeddable and four small changes away from being a pleasure to embed.

The single blocker is not performance, not correctness, not dependencies — it is the output contract. The library emits its own result types (types.Result, types.ReportData) that no downstream ecosystem speaks. Every embedder pays a bespoke adapter tax. Make the result interoperable and the library becomes the obvious choice for anyone who needs "validate the code blocks in these docs" without shelling out to a binary.

The bottom line A real, high-stakes consumer (BuildFlow) studied integrating this library in-process, found the engine fit perfectly, and stopped at the output-model mismatch + an unconditional tree-sitter dependency. Both are cheap to fix. Fix them and the library gains a first-class embedder overnight.

Method: full read of pkg/validator.go, pkg/extractor.go, pkg/types/, pkg/languages/, and go.mod; cross-checked against a concrete integration study (BuildFlow's in-process SDK-checker pattern). This report is intentionally narrow: it does not revisit CLI accuracy or false-positive strategy coverage — see the sibling make-it-superb.html for that.

◆What's already excellent (keep all of this)

Before the criticism: the foundation a library consumer wants is already here. None of the proposed changes should compromise these.

+
Invariant-enforcing Result constructors StatusError ⇔ Error != nil is enforced at construction time via newResultWithStatusAndError (panics on violation). This is exactly how a library makes illegal states unrepresentable. A consumer never has to nil-check a status/error mismatch.
+
Pluggable languages.Validator + Registry The Validator interface (Language(), Validate(ctx, code), IsAvailable()) with DefaultRegistry() means an embedder can register their own validator or swap the registry wholesale via WithRegistry(). This is the right extension point.
+
Branded types throughout FileID, LineNumber, BlockIndex, Language, ErrorCode — all branded strings/uints with Validate(). A consumer cannot accidentally pass a block index where a line number belongs. This discipline is rare and valuable.
+
Pure Go, no CGO, context-aware The whole reason an embedder reaches for this over tree-sitter-cli is "one binary, no native deps, cross-compiles anywhere." Validate() takes a context.Context so a host can cancel. Both are essential for embedding.
+
Functional-options builder on FileValidator New(verbose).WithLanguages(…).WithConcurrency(n).WithMaxBlocks(n) is the idiomatic Go API style. Additive, zero booleans, readable at the call site.

01The one blocker: the result speaks a private dialect

Every other finding in this report is workable around. This one is structural. A library's output type is its most-imported surface — more code depends on it than on any function. types.Result and types.ReportData are clean, but they are md-go-validator's own vocabulary. No editor, no SARIF emitter, no LSP server, no finding-aggregator speaks them natively.

Blocking · high impact

Result has no neutral interchange target

pkg/types/result.go · pkg/types/report.go

A consumer that wants SARIF, JSON-LSP, or a unified finding pipeline must hand-write a converter: Result{Status:Error} → their finding type, field by field, including the ErrorCode enum mapping. That converter is bespoke per consumer, drifts as Result evolves, and is the kind of glue that makes maintainers say "let's just shell out to the CLI instead." The CLI at least hands you a documented JSON schema.

The irony: the project already depends on go-output for rendering. The sibling ecosystem go-finding (a typed Finding with Position{File,Line,Column}, Severity, RuleName) is the natural neutral contract and is what every modern Go linter emits. Adopting it — or any documented interchange struct — turns every embedder's adapter from ~30 lines into a one-liner. 

// What every embedder writes today (~30 lines, per-consumer, drift-prone):
func resultToFindings(toolName string, results []mdgovalidator.Result) Report {
    var findings []Finding
    for _, r := range results {
        if r.Status != StatusError { continue }
        findings = append(findings, Finding{
            Tool: toolName,
            File: string(r.File),
            Line: int(r.LineNumber),
            Message: r.ErrorMessage(),
            Code:   r.ErrorCode.String(),
        })
    }
    return Report{Findings: findings}
}

// What it should be:
report := validator.ValidateDirectory(ctx, dir)
// report already IS a []Finding (or emits them via a callback).
Fix Expose a results → neutral-findings conversion in the library, not in every consumer. Two options: (a) add a finding adapter sub-package that converts []Result → []Finding; or (b) have ValidateDirectory accept an Emitter callback (see the proposal section). Either way, the library owns the mapping once.

02Library findings (ordered by embedder impact)

Important · dep-graph blocker

gotreesitter is an unconditional transitive dependency

pkg/languages/treesitter_validator.go:7-8 · imported at package level

pkg/languages imports gotreesitter and gotreesitter/grammars at package scope. Because FileValidator and the Language type live in pkg/languages, any embedder that only ever validates Go blocks (stdlib-only path) still pulls in all six tree-sitter grammars — several megabytes of embedded parser data and non-trivial compile time. An embedder that cares about binary size or a lean dependency graph (BuildFlow has zero tree-sitter deps today) hits this immediately.

Fix Move the tree-sitter validator into its own importable sub-package (pkg/languages/treesitter) and have DefaultRegistry() register it only when that package is imported (blank-import or an explicit RegisterTreeSitter(r) call). Go-only embedders then pay zero grammar cost; multi-language embedders add one import line.
Important · host cannot push config

No exclude / filter API for library callers

pkg/validator.go · FileValidator has no WithExclude()

FileValidator has WithMaxFiles, WithMaxBlocks, WithConcurrency, WithLanguages, WithRegistry — but no way to say "skip vendor/, node_modules/, generated docs." collectSupportedFiles only skips a hard-coded shouldSkipDir set. A host (BuildFlow, an editor) almost always has its own exclude config it wants to push down. Today it can only post-filter results, which means the validator still reads and parses files the host will throw away.

Fix Add WithExcludePatterns([]string) (doublestar globs) and/or WithFileFilter(func(path string) bool). The CLI's missing exclude support (CONSUMER_PERSPECTIVE.md #4) is the CLI symptom of this same library gap.
Important · loses error context

Errors wrap with fmt.Errorf, not typed/sentinel

pkg/validator.go, pkg/extractor.go — throughout

The library wraps errors as fmt.Errorf("context: %w", err) everywhere. For a library consumer that wants to branch on failure class — "is this a cancelled run, a missing file, a parse failure, or an unavailable validator?" — there are exported sentinel errors (errPathEmpty, errNoValidatorForLang) but they are unexported, so errors.Is from outside the package cannot match them. A host is reduced to string-matching.

Fix Export the sentinel errors (ErrPathEmpty, ErrNoValidatorForLang, ErrPathNullByte) so embedders can do errors.Is(err, mdgovalidator.ErrNoValidatorForLang). Keep fmt.Errorf wrapping (your wrapcheck linter is right to require it); just make the targets referenceable.
Important · buffering at scale

No streaming / callback API for large doc sets

pkg/validator.go · ValidateDirectory returns []Result fully buffered

Internally there is a worker pool (processFilesParallel), but the public API buffers every result into one []Result slice before returning. For a host validating a monorepo's worth of docs, that means holding all results in memory and getting no feedback until the end. An embedder that wants a progress bar, an early-abort on first-error, or incremental UI updates has no hook.

Fix Add a variant that emits incrementally: ValidateDirectoryFunc(ctx, dir, func(Result) error) error. The host returns a non-nil error to stop early. Trivially implementable on top of the existing worker pool by forwarding results instead of appending them.
Nice to have · version coupling

Pinned go-output v0.17.2 couples embedders to a stale line

go.mod

go-output is used only by pkg/output (CLI rendering). But because it is a direct require, an embedder on a newer go-output (e.g. v0.18/v0.19) gets MVS resolution that the library hasn't been validated against. For a library whose embedding surface shouldn't need a renderer at all, this is needless coupling.

Fix Move pkg/output behind the CLI boundary so library importers don't transitively require go-output. Combined with the tree-sitter split, the library's true dependency footprint shrinks to "Go stdlib + your branded-id helper."

03What the target API could look like

If the four findings were addressed, embedding md-go-validator into any Go tool becomes a one-screen integration. Here is the shape a happy embedder would write.

Before — what an embedder writes today

// Embedder must: build the validator, convert results, filter excludes
// itself, swallow unexported errors, and accept the tree-sitter binary tax.
v := mdgovalidator.New(false).
    WithLanguages([]languages.Language{languages.LangGo})

results, err := v.ValidateDirectory(ctx, docsDir)
if err != nil { // cannot errors.Is() any sentinel — they're unexported
    return err
}
// now hand-convert results → host's finding type (~30 lines, per host)
for _, r := range results {
    if isUnderExcludedDir(string(r.File)) { continue } // host re-filters
    if r.HasError() { emit(convert(r)) }
}

After — the target state

// One import for Go-only embedders (no tree-sitter, no go-output):
import mdgovalidator "github.com/larsartmann/md-go-validator/pkg"
import "github.com/larsartmann/md-go-validator/pkg/finding" // adapter

v := mdgovalidator.New(false).
    WithLanguages(languages.Only(languages.LangGo)).
    WithExcludePatterns([]string{"vendor/**", "node_modules/**"})

// Streaming + neutral contract in one call:
return v.ValidateDirectoryFunc(ctx, docsDir, func(r mdgovalidator.Result) error {
    if r.HasError() {
        emit(finding.FromResult("md-codeblock", r)) // one-liner
    }
    return nil // return err to abort early
})
This is the target Every line an embedder writes now lives in the embedder's domain; zero lines are spent on format conversion, exclude re-implementation, or error classification. That is what "obvious choice to embed" looks like.

Minimal: if only one thing ships

If only the output-contract fix ships, the integration still collapses from ~30 lines of glue to ~3. That single change carries most of the value of this entire report.

04Roadmap to "obvious to embed"

1

Ship a finding adapter (the 80%-value step)

Add pkg/finding exposing FromResult(tool string, r Result) Finding and FromResults(tool string, rs []Result) []Finding against a typed Finding (go-finding or an equivalent neutral struct). Document it as the canonical way to consume results. This alone unblocks most embedders.

2

Split tree-sitter into an opt-in sub-package

Move treesitter_validator.go to pkg/languages/treesitter. DefaultRegistry() registers Go only; multi-language embedders add a blank import or explicit RegisterTreeSitter(r). Go-only embedders lose the grammar binary cost entirely.

3

Export sentinels + add WithExcludePatterns

Export ErrPathEmpty, ErrNoValidatorForLang, ErrPathNullByte for errors.Is. Add WithExcludePatterns([]string) / WithFileFilter(func(string) bool) so a host pushes its own skip config instead of post-filtering.

4

Add streaming + decouple the renderer

Add ValidateDirectoryFunc(ctx, dir, func(Result) error) error. Move pkg/output (go-output) behind the CLI boundary so library importers don't transitively require a renderer. Publish an API-stability note for pkg/.

05What NOT to do

  • Don't weaken the invariant-enforcing constructors. newResultWithStatusAndError panicking on bad state is a feature, not a bug. Embedders rely on the StatusError ⇔ Error guarantee. The neutral-finding adapter must preserve it, not paper over it.
  • Don't make Result an interface. The concrete struct with branded fields is exactly what makes it cheap to convert and impossible to misuse. A "Resultable" interface would add ceremony without value.
  • Don't hide tree-sitter behind a build tag. Build tags fragment the build matrix and break go install. A sub-package + explicit registration is the Go-idiomatic way to make a heavy dependency opt-in.
  • Don't ship the adapter as a separate module prematurely. Keep pkg/finding in-repo until there's a real second consumer. Premature module splits create replace-directive churn (BuildFlow's go-work experience is a cautionary tale).
  • Don't conflate "validates" with "embeddable." The engine validating correctly is necessary but not sufficient. A library is embeddable when the output contract, dependency footprint, and error model all match the embedder's world — none of which the parser's correctness tells you anything about.
