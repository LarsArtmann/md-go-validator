md-go-validator needs better errors, not just fewer errors

A real-world field report from running md-go-validator against the gogenfilter library (v3.2.0, 71 code blocks, 9 failures). The tool's architecture is sound — but its error messages lie, its escape hatch is hidden, and its CLI surface is too thin. Here is what to fix and in what order.

00Summary

The validator ran on gogenfilter's full documentation tree (71 Go code blocks across README.md, internal planning docs, and research notes). It found 9 failures. After manual triage: 1 was a real README bug (a stray import mixing package-level and statement-level code), and 8 were illustrative snippets that are intentionally not valid Go (anti-pattern comparisons, API sketches, pseudo-code).

The tool correctly identified that something was wrong with all 9. The problem is what happens next: the error messages are actively misleading, the skip directive escape hatch is buried, and there is no config file to scope or customize behavior per project.

The bottom line The architecture is good. The parsing strategies are clever. But when the tool fails, it reports the wrong error from the wrong strategy — and gives the user no hint about how to proceed. Fix the error diagnostics first; everything else is secondary.

◆What already works

Before the criticism: this tool has genuinely good bones. The multi-strategy cascade is the right idea, and the non-Go language support via tree-sitter is a real differentiator.

+
Five-strategy cascade is architecturally sound Complete file → package wrapper → function wrapper → expression → statements. This is the right way to handle documentation snippets, which are rarely complete translation units.
+
Configurable skip directives in the library API ExtractCodeBlocksWithConfig accepts a custom SkipDirectivesConfig. The library API is more flexible than the CLI exposes. Six default directives cover the common cases.
+
Clean exit codes for CI 0 = success, 1 = validation errors, 2 = tool errors. Correctly distinguishes "your docs have problems" from "the tool crashed." This matters for CI gating.
+
Multi-language via embedded tree-sitter Go, TypeScript, TSX, Rust, Nix, HCL, Terraform, Templ — all in one binary, no CGO, no external dependencies. This is a real competitive advantage over tools that only handle Go.
+
Testable CLI architecture osExit is a mockable package var, runWithConfig returns an int instead of calling os.Exit directly. Shows forethought about testability.

01Findings

Ordered by impact on the consumer experience.

Blocking · #1 usability problem

Error messages always report the wrong strategy's error

pkg/languages/go_validator.go:65-97 · createValidationError()

When all five strategies fail, createValidationError re-parses the code with strategy 1 (raw) to extract the error for the user. But the code already failed strategy 1 — that is why strategies 2-5 ran. The error the user sees is always from the least informative strategy, never the one that got closest.

Real example from gogenfilter. A snippet mixing import and statements reports: 

// What the user sees — from strategy 1 (raw parse):
1:1: Go syntax error: expected 'package', found 'import'

// What actually happened — strategy 3 (func wrapper) got further:
// It failed at line 2, not line 1 — because import can't go
// inside a function. THAT error is the useful one.
Fix: track the best attempt Run all five strategies, capture each one's error, and report the one from the strategy that parsed the furthest (highest error line number). That is the strategy that got closest to succeeding — its error is the most informative.
Blocking · no diagnostic for the #1 pattern

Mixed-scope snippets fail with zero guidance

pkg/languages/go_validator.go:25-62 · Validate()

The dominant documentation pattern is "show the import, then show the usage." This snippet shape has import declarations at the top and statements below — it belongs in two scopes (package level + function body). No single strategy can wrap it, because:

  • Strategy 2 (package wrapper) fails — statements can't live at package level
  • Strategy 5 (func wrapper) fails — import can't live inside a func

The two strategies that could help are mutually exclusive for this shape. The tool gives no hint that this is the problem — just a generic syntax error.

Fix: detect and name the pattern After all strategies fail, scan for the mixed-scope signature (has import/type/func decls AND bare statements). If found, emit a targeted message: "snippet mixes package-level declarations with function-body statements — add // skip-validate if illustrative, or split into two blocks."
Important · escape hatch is hidden

Skip directives are documented but never suggested on failure

pkg/extractor.go:121-129 · --help output · README

The skip directive system (<!-- skip-validate -->, // skip-validate, //nolint, etc.) is a well-designed escape hatch. But it is only visible if the user reads --help or the README. When a block fails, the error message does not mention that skip directives exist. A new user staring at expected 'package', found 'import' has no idea there is a way to mark a block as intentionally illustrative.

Fix: suggest the escape hatch in errors Append a hint to every validation error: "If this snippet is illustrative, add // skip-validate as the first line inside the code block." One line of output that saves every new user a trip to the README.
Important · no project-level config

No config file — everything is CLI flags

cmd/md-go-validator/main.go · CONSUMER_PERSPECTIVE.md #2

A real project needs to persist: which languages to validate, which directories to include/exclude, custom skip directives, output format. Today this means a long CLI invocation every time, or a shell script wrapper. The library API already supports custom skip directives (ExtractCodeBlocksWithConfig) but the CLI never wires it through.

The project's own CONSUMER_PERSPECTIVE.md already identifies this as P0 gap #2. My field experience confirms: the first thing I wanted after the initial run was a way to say "skip docs/research/** and use a custom skip directive."

Fix: add .md-go-validator.yaml Support a config file with languages, exclude globs, skip-directives, and format. Add --init to scaffold it. This is the single biggest adoption unlock.
Important · feature exists but is unreachable

CLI never exposes custom skip directives

cmd/md-go-validator/main.go vs pkg/extractor.go:148-158

The library API has ExtractCodeBlocksWithConfig(content, langs, customDirectives) — a clean, tested feature for project-specific skip markers. But the CLI always calls ExtractCodeBlocks, which hardcodes DefaultSkipDirectives(). A CLI user cannot add <!-- sketch --> or // example as custom skip directives without writing Go code against the library.

Fix: --skip-directive flag --skip-directive "<!-- sketch -->" (repeatable) that feeds into the config. One-line wiring through an existing, tested code path.
Important · directory excludes are hardcoded

shouldSkipDir is not configurable

pkg/validator.go:473-486

shouldSkipDir hardcodes six names: dot-dirs, node_modules, vendor, build, dist. A Go project cannot exclude testdata/, reports/, or docs/planning/** without scoping the CLI path argument manually. The only workaround is md-go-validator website/src/content/docs/ — which is what gogenfilter's CI actually does.

Fix: --exclude flag or config exclude list --exclude "docs/research/**" (repeatable glob) that augments the default skip list. Trivial to implement — filepath.WalkDir already visits every dir.
Nice to have · documentation accuracy

README architecture diagram claims features that don't exist

README.md:262-263, 297

The architecture diagram lists "External cmds" and "Custom" as validator types. The README's "Future Enhancements" lists "Custom validators." But all validators are embedded tree-sitter grammars — no external command support or custom validator plugin system exists in the code. This oversells the current state and misleads potential adopters.

Fix: correct the diagram Remove "External cmds" and "Custom" from the architecture diagram, or label them explicitly as "planned." Honesty about scope builds trust.
Nice to have · minor waste

parser.AllErrors requested but only first error reported

pkg/code/util.go:31-38 · go_validator.go:77-89

ParseGo uses parser.AllErrors flag (requests all parse errors from go/parser), but createValidationError only extracts the first error from scanner.ErrorList. The AllErrors flag is wasted work — parsing collects errors that are never shown. Not wrong (first error is usually the root cause), but the flag should either be removed or the additional errors should be surfaced as context.

02Deep dive: the error quality problem

This is the single highest-leverage fix because it affects every consumer, on every run, not just the false-positive cases. Even when the validator correctly identifies a broken block, the error message points the user in the wrong direction.

The mechanism

Validate() tries strategies 1-5 sequentially. When all fail, it calls createValidationError(fset, code), which re-parses with strategy 1 (raw code) to extract the error: 

// go_validator.go:65-67 — always strategy 1
func (v *GoValidator) createValidationError(fset *token.FileSet, code string) error {
    err := codeutil.ParseGo(fset, code)  // ← always raw parse
    if err == nil {
        return nil
    }

Why this is wrong

Strategy 1 is the most restrictive — it expects a complete Go file. Every documentation snippet that needs wrapping (strategies 2-5) will fail strategy 1 at line 1. So the error the user sees is always expected 'package', found '<first token>' — which tells them nothing about why their snippet is invalid, only that it doesn't start with package.

What the error should be

The most useful error comes from the strategy that parsed the furthest before failing — that strategy's error is closest to the actual problem:

Current (always strategy 1)
Proposed (best attempt)
# Snippet: import + statements
1:1: Go syntax error:
  expected 'package', found 'import'
# From strategy 3 (func wrapper)
2:1: Go syntax error:
  expected '}', found 'result'

hint: snippet mixes package-level
and function-body code
Implementation sketch Track (strategy, error) pairs for all five attempts. After all fail, select the error with the highest Pos.Line. If the best error is from strategy 2 or 3 and the snippet contains both import/type and bare statements, append the mixed-scope hint. ~30-50 lines in one function.

03Deep dive: skip directive discoverability

The skip directive system is well-designed — it supports six markers, can be placed before or inside a block, and is configurable in the library API. The problem is discoverability. Here is the journey a new user takes today:

Run validator → 9 errors → Confusing messages → Read --help? → Find skip directives

The last two steps are optional and most users won't take them. They will either ignore the errors (if the tool isn't in CI) or remove the tool from CI (if it is). The fix is trivial: mention skip directives in every error message.

Before/after: the error output

Proposed error output

  found 'import'

  Code:
    1 | import "errors"
    2 |
    3 | root, err := ...
README.md:231 (block #6)

  found 'result' (strategy: func wrapper)

  This snippet mixes package-level declarations
  with function-body statements.

  If illustrative, add as first line:
    // skip-validate
The principle Escape hatches only work if users know they exist. The error message is the moment of maximum need — that is when and where to surface the solution.

04The config file gap

gogenfilter's CI runs md-go-validator -f table website/src/content/docs/ — it scopes to a specific directory because there is no way to exclude docs/research/** or docs/planning/**. When I ran the validator on the full project, I got 9 errors that CI never sees. A config file would solve this: 

# .md-go-validator.yaml
languages: [go, typescript]

exclude:
  - "docs/research/**"
  - "docs/planning/**"
  - "docs/status/archive/**"

skip-directives:
  - "<!-- sketch -->"
  - "// pseudo-code"

format: table

This makes md-go-validator . "just work" on any project. The library API already supports custom skip directives and the registry already supports language selection — the config file just needs to wire them into the CLI.

Implementation scope
  • Parse .md-go-validator.yaml (or .json) from CWD
  • CLI flags override config file values (standard precedence)
  • --init writes a default config with commented examples
  • Wire config's skip-directives into the existing ExtractCodeBlocksWithConfig path
  • Wire config's exclude into a new filter inside collectSupportedFiles

05Pareto roadmap

Ordered so the first step delivers the largest improvement in perceived quality. The companion report (2026-06-25_product-feedback-make-it-superb.html) covers the strategy-coverage gap (imports+statements, ellipsis). This roadmap focuses on the error quality and adoption ergonomics that complement those fixes.

1

Fix error diagnostics (best-attempt reporting)

Track all five strategy errors, report the one that parsed furthest. Add mixed-scope detection. Add skip-directive hint to every error. ~50 lines in go_validator.go. This alone transforms every error message from misleading to helpful.

2

Add config file + --init

.md-go-validator.yaml with languages, exclude, skip-directives, format. Wire exclude into directory walking. Wire skip-directives into the existing ExtractCodeBlocksWithConfig path. Enables per-project customization without Go code.

3

Expose custom skip directives via CLI

--skip-directive "<!-- sketch -->" (repeatable). One-line wiring through an existing, tested library API. Low effort, high flexibility.

4

Add --exclude flag for directory globs

--exclude "docs/research/**" (repeatable). Augments the hardcoded shouldSkipDir list. Makes full-project validation usable without path scoping tricks.

5

Correct the README architecture diagram

Remove "External cmds" and "Custom" from the validator diagram, or mark them as planned. Two lines of markdown. Eliminates false expectations.

6

Publish a GitHub Action

uses: LarsArtmann/md-go-validator@v1 instead of a 4-step YAML block. The companion report identifies this as the highest-leverage distribution feature. Becomes viable once the error quality and config file are in place.
