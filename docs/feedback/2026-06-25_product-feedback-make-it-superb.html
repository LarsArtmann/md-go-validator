01 The Verdict

The tool has a solid foundation — clean architecture, branded types, functional options, tree-sitter for non-Go languages, structured exit codes, JSON/YAML output. As engineering it's well built. As a product it has one fatal flaw that sits between the tool and every consumer:

The core problem Run on a real Go library's documentation, the validator reports 130 errors and finds zero real bugs — a 100% false-positive rate. The five "parsing strategies" don't cover the three documentation conventions that real technical writers use every day. A tool that cries wolf this loudly cannot live in CI; contributors will mute it within a week.

The good news: the gap is narrow and entirely fixable. This is not a redesign — it's filling a well-understood coverage hole and adding the adoption ergonomics the project's own CONSUMER_PERSPECTIVE.md already identifies.

For contrast, the same repo's own lightweight cmd/doc-check tool (which only verifies import paths resolve and pkg.Symbol references exist) reports "All 445 references valid" — it catches the bug class that actually hurts consumers (renamed/removed symbols, broken imports) with zero noise.

The three patterns, in code

Every one of the 130 failures is one of these:

Pattern A — Imports + statements (68 cases)

// The single most common documentation pattern.
// Shows the import, then the usage. NO package decl, and a top-level
// statement — which every wrap strategy mishandles.
import "github.com/acme/lib/v3"

b, _ := lib.New()
defer b.Close()

Strategy 2 (package main + code) fails because a top-level statement can't live outside a function. Strategy 5 (wrap in func _()) fails because import can't live inside a function. The two strategies are mutually exclusive for this shape.

Pattern B — Ellipsis placeholders (57 cases)

// "Implementation elided" — universally understood by readers.
func NewStore(db *sql.DB, opts ...Option) *Store { ... }
type serviceBuilder struct { ... }

Pattern C — Pseudo module files (5 cases)

require (
    github.com/acme/core v1.6.0
)
replace github.com/acme/core => ../core

03 Root Cause — A Strategy Coverage Gap

The architectural insight The Go validator's five wrap strategies (pkg/languages/go_validator.go) cover two buckets — declarations only or statements only — but never imports + statements together, which is the dominant documentation shape. The strategies are tried sequentially; none of them can split an import block from a statement block and reassemble them correctly.

The consequence is a tool that is strict about the wrong thing:

Validator does this But documentation is…
Demand a compilable translation unit Prose-as-code — illustrative, deliberately incomplete
Treat ... as a syntax error A universal idiom for "body omitted"
Reject pseudo-signatures The most readable way to show an API surface

This is a category error: documentation snippets are not programs. A useful validator must meet docs where they are, not where the Go parser wishes they were.

05 The Pareto Roadmap

Ordered so the first step delivers the largest swing in perceived quality. The single highest-leverage change is accuracy — no amount of polish rescues a tool that's wrong 100% of the time.

1% → 51% — Make it right before you make it shiny

  • Fix the strategy coverage gap (G1). Add a 6th strategy that splits the snippet into a leading import block and a trailing statement block, wraps the imports at package level and the statements inside func main(). This alone clears ~68/130 false positives.
  • Recognize elision (G2). Before parsing, detect { ... }, {...}, and bare trailing ... as "body omitted." Either skip these snippets, or normalize { ... } → {} before parsing. Clears ~57/130.

4% → 64% — Make it adoptable

  • Config file + --init (A2). A committed .md-go-validator.yaml so md-go-validator . "just works" with per-repo settings.
  • Exclude patterns (A3). Glob ignore for vendor/, node_modules/, generated docs. Required for any real monorepo.
  • --baseline regression mode (G4). Accept a snapshot of known errors; fail only on new ones. This is what lets a noisy repo adopt the tool today and ratchet down over time.

20% → 80% — Make it the obvious choice

  • Reusable GitHub Action (A1). uses: LarsArtmann/md-go-validator@v1 instead of a 4-step YAML block. Highest-leverage distribution feature.
  • Pre-commit hook (A4). One .pre-commit-hooks.yaml opens the entire pre-commit.com ecosystem.
  • Strict --fail-on-skipped (A5). Let teams that want zero escape-hatches enforce it.

06 Concrete Fixes — Ready to Implement

Fix 1: A pre-parse normalizer for elision

Cheapest, highest-impact. Before handing code to the 5 strategies, normalize the three documentation idioms. This is a pure-function string transform, trivially unit-testable. 

// Pseudocode for code/normalize.go
func NormalizeDocIdioms(code string) string {
    // 1. "body omitted" — { ... } → {}
    code = rxBodyOmitted.ReplaceAllString(code, "{}")
    // 2. trailing "..." on its own line or as a token → drop the line
    code = rxElidedLine.ReplaceAllString(code, "")
    // 3. pseudo go.mod content → caller may short-circuit (skip)
    return code
}
Why this works Normalizing { ... } → {} turns every API-surface snippet into valid Go that the existing strategies 2/3 already accept. It costs one regex pass and removes an entire false-positive category with zero new parser logic.

Fix 2: A 6th strategy for import + statement snippets

// Strategy 6 — split imports from statements, wrap each correctly.
func wrapImportsAndStatements(code string) string {
    // Hoist every `import (...)` / `import "..."` line to the top.
    importBlock, rest = splitImports(code)
    return "package main\n\n" + importBlock +
           "\nfunc main() {\n" + indent(rest) + "\n}"
}

Add this as strategy 6, before falling through to the error path. It handles the dominant "show the import, show the usage" pattern that no current strategy can.

Fix 3: --baseline for incremental adoption

# Generate the known-errors snapshot once
md-go-validator -f json . | jq '.errors[].file + ":" + (.line|tostring)' > .md-go-baseline.txt

# CI then fails only on NEW errors
md-go-validator --baseline .md-go-baseline.txt .

This converts "130 errors, I give up" into "0 new errors, ship it" — and lets teams ratchet down the baseline as they fix real docs over time.

07 What NOT to Do

  • Don't ship to CI before fixing accuracy. A 100% false-positive gate teaches contributors to add <!-- skip-validate --> everywhere or remove the check. Trust, once lost, is not regained.
  • Don't make ... a hard error with no escape. Elision is a documentation feature, not a defect. The skip directive is a last resort, not the primary UX.
  • Don't conflate "parses" with "valuable." The competing cmd/doc-check proves that symbol-resolution (does pkg.Foo exist?) catches real consumer-breaking bugs that pure syntax never will. Consider a future "reference check" mode as a complement, not a replacement.

08 North Star

The one-sentence goal A consumer should be able to drop md-go-validator into a mature repo's CI, see either a clean pass or a short list of genuinely broken blocks, and trust that the next failure they see is a real defect — never noise.

Today that sentence is false: the tool reports 130 errors and zero of them matter. The gap between here and there is two regexes, one extra parse strategy, a config file, and a GitHub Action. None of it requires a rethink — it requires finishing the job the architecture was already built to support.

The foundation (pure-Go, multi-language tree-sitter, clean registry, structured output) is genuinely good. The remaining work is the difference between a tool that checks syntax and a tool people actually use.
