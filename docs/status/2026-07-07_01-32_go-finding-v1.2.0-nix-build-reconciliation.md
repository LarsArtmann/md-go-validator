## d) TOTALLY FUCKED UP 💥

**Turn 1 declared "done" prematurely — I created a build-path split-brain.**

In turn 1 I:

1. Saw the typecheck error (`cannot use r.File.String() as FilePath`).
2. Patched `pkg/finding/finding.go`.
3. Ran only `go build ./...` and `go test ./...`.
4. Saw green and said _"Build and tests pass."_

What I **missed**: the nix build was **never run**, and it is the **authoritative** build path for this project. Because `package.nix` injects a `replace github.com/larsartmann/go-finding => <flake-input-src>` via `postPatch`, the nix build compiles against **whatever version the `go-finding-src` flake input points to** — NOT the version in `go.mod`. The flake input was still pinned to **v1.0.0**, whose `Position.File` is a plain `string`. So:

- Turn 1 `go build` (uses go.mod v1.2.0): the `finding.FilePath(...)` cast was **correct** → green.
- Turn 2 `nix build` (uses flake input v1.0.0 via replace): the same cast was **wrong** — _"cannot use FilePath as string"_ → red.

I fixed a symptom in turn 1, broke the nix build in doing so, and only noticed in turn 2 when the user re-ran buildflow. The **single correct fix** was a coordinated update of **Go code + flake input + flake.lock together**, which I did across two turns because I never read `flake.nix`/`package.nix` and never ran `nix build` in turn 1.

**Root behavioral error:** I used the non-authoritative build path and skipped the project's canonical `nix build` / `nix flake check` — despite `AGENTS.md` listing them as the primary commands.

## e) WHAT WE SHOULD IMPROVE 🛠️

### Process

1. **Always run `nix build` / `nix flake check`, not just `go build`.** This project's source of truth is the nix build (the `postPatch` replace proves it). `go build` alone is insufficient and can give false green.
2. **Detect version skew proactively.** When `go.mod` pins a dependency at vX and a flake input pins the same dependency at vY with a `replace` override, that is a **latent bug**. The `go-finding-src` input must stay in lockstep with `go.mod`'s `go-finding` line — or the replace mechanism should be removed.
3. **Read `flake.nix` + `package.nix` before editing Go that touches a replace-overridden dependency.** The replace directive means the Go API you compile against locally may differ from what nix compiles against.
4. **Document the replace mechanism in `AGENTS.md`.** This is a non-obvious, hard-to-discover gotcha exactly the kind of thing AGENTS.md exists for. It belongs under a "Nix Gotchas" subsection. **`[✅ RESOLVED]`** — Added "Nix Gotcha: go-finding-src replace directive" subsection to `AGENTS.md:189`.

### Architecture

5. **Reconsider the `postPatch` replace for `go-finding`.** `go-finding` v1.2.0 is a published, tagged, proxy-available module. A `replace` directive forcing every bump to be a 3-place update (go.mod + flake input + flake.lock) is friction that **just caused this bug**. If the replace exists only for pre-publish local iteration, it should be dropped now that v1.2.0 is on the proxy. _(See §g.)_ **`[🔵 BLOCKED]`** — Awaiting user design decision; tracked in `TODO_LIST.md`.
6. **Single source of truth for dependency versions.** Today: `go.mod` says v1.2.0, flake input says (was) v1.0.0. Make one canonical; ideally just `go.mod`. **`[🔵 BLOCKED]`** — Same as #5; awaiting the postPatch decision.

### Understanding gap to close

7. **The `vendorHash` did not change** between v1.0.0 and v1.2.0 replace targets. My hypothesis: with `proxyVendor = true`, the `go-modules` FOD hashes only **proxy-fetched** deps, and the replace target (a nix store path) is injected at the **build** step, not counted in the FOD hash. I have not rigorously confirmed this. Worth verifying so future bumpers aren't confused when the hash stays stable across a version that clearly changed source. **`[🟡 OPEN]`** — Tracked in `TODO_LIST.md` ("Verify vendorHash stability hypothesis").

## g) Top #1 Question I Cannot Answer Myself ❓

**Why does `package.nix` use a `postPatch` `replace` directive to override `go-finding` with the `go-finding-src` flake input — and should it be removed now that v1.2.0 is published on the Go proxy?**

`go-finding` v1.2.0 is a tagged, proxy-available module (`go.mod` references it directly and `go build` resolves it fine). Yet `package.nix` appends `replace github.com/larsartmann/go-finding => <flake-input-store-path>`, which means:

- The nix build **ignores** `go.mod`'s `go-finding` version.
- Every `go-finding` bump requires updating **three** places in lockstep (`go.mod`, `flake.nix` input, `flake.lock`), and forgetting any one of them produces exactly the split-brain failure we just spent two turns debugging.

I cannot tell whether this `replace` is:

- **(A) Intentional and still needed** — e.g., you iterate on `go-finding` locally and publish _after_ validating against `md-go-validator`, so the replace lets you test unpublished source; **or**
- **(B) Legacy** — left over from before v1.2.0 was published, now pure overhead and a bug magnet.

If **(B)**, I recommend deleting the `postPatch` block, the `go-finding-src` input, and the `package.nix` `go-finding-src` parameter — collapsing to a single source of truth (`go.mod`). If **(A)**, I recommend at minimum adding the drift guard (item #7) and documenting the coupling in `AGENTS.md` (item #6).

This is a design decision only you can make.
