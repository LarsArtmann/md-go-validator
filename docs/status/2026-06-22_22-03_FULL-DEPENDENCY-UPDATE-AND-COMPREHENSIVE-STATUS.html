DTotally Fucked Up

Recurring footgun (3rd strike)

The vendorHash drifts silently and breaks the build on every dep upgrade

package.nix — buildGoModule fixed-output derivation

buildGoModule pins a vendorHash in package.nix. Whenever go.sum changes (any dep bump), the hash goes stale and nix build .# fails with hash mismatch. This has now bitten the project three times:

  • e0aeb59 — "update stale nix vendor hash after go-output upgrade"
  • 0fc0f12 — "upgrade go-output … and fix Nix version attribute"
  • This session — go-output v0.17.0 → v0.17.2 broke the build again

The failure mode is also misleading: buildflow update reports green (it updated go.mod) while nix build is red. A maintainer trusting the "update succeeded" message ships a broken build. BuildFlow has no post-update step that syncs the Nix vendor hash, so the dependency-update workflow is structurally incomplete.

Fix direction: Make the hash self-healing. Options: (a) vendorHash = lib.fakeHash during dev + a CI check that asserts the real hash is committed; (b) a buildflow post-update hook running nix build .# and patching package.nix; (c) a nix run-able updater script. The exact choice needs a human decision (see Question G).
Process observation

"BuildFlow update passed" is not "everything is updated"

buildflow update — 2 passed / 2 skipped

buildflow update ran 4 sub-steps: cargo-update and npm-update were correctly skipped (no Rust/JS in this repo), and go-mod-update + nix-flake-update passed. But "passed" only means the step executed — it does not mean the build still works afterwards, nor that the vendorHash was synced, nor that all inputs actually moved. The green summary masked the broken Nix build until I ran nix build .# manually. Trusting the top-line summary would have shipped breakage.

Lesson: "Update ran" ≠ "Update is verified". Always run the build after a dep bump, and treat the absence of a verification step as a gap to fill.

EWhat We Should Improve

1. Make the dependency-update workflow self-verifying

The update flow currently stops at "go.mod changed". It must continue through "vendorHash synced" and "build green". A single missing link (the Nix hash) has now caused three broken builds. This is the highest-impact process fix available.

2. Make the go-error-family decision explicit (and stop the persistent red)

92 hierarchical-errors findings (45 generic_return) will keep BuildFlow red until either go-error-family is adopted or the gate is explicitly disabled via ADR. A quality gate that is perpetually red is worse than no gate — it trains everyone to ignore failures. Decide and document.

3. Populate the missing standard doc files

FEATURES.md, TODO_LIST.md, ROADMAP.md are absent; docs/DOMAIN_LANGUAGE.md is a placeholder. These are defined in the project documentation standards as required files with distinct purposes. Their absence means project state is invisible without reading code, and every status report (this is #23) re-derives the same information.

4. Eliminate global mutation in cmd/main.go

cmd/md-go-validator/main.go uses package-level globals (version, osExit, os.Stdout) that tests must mutate. This forces serial tests and //nolint:paralleltest hacks, and was the root cause of the data race fixed in e0aeb59. The proper fix is dependency injection.

5. Add a post-update verification to CI

Even without buildflow changes, a CI job that runs nix build .# on dep-bump PRs would catch vendorHash drift before merge. The dogfooding + nix flake check jobs exist; a dep-change verification step does not.

Option Mechanism Tradeoff
A — buildflow post-update hook A hook runs nix build .# after go-mod-update and rewrites package.nix with the new hash Fixes it at the source, but couples the build tool to Nix internals; depends on buildflow supporting such a hook
B — CI verification job A CI step asserts nix build .# is green on any PR touching go.mod/go.sum Catches drift before merge but does not auto-fix — still requires a manual hash bump commit
C — lib.fakeHash dev workflow Use a fake hash in dev; CI asserts the real hash is committed Fastest dev loop, but changes the contribution flow and risks shipping a fake hash
D — updater script (nix run-able) A small update.nix / shell script maintainers run after dep bumps Explicit and simple, but only works if people remember to run it (same class of footgun, smaller blast radius)
