## e) WHAT WE SHOULD IMPROVE

### 1. Architecture & Type Model

| Opportunity                     | Current State                                                                 | Proposed Improvement                                                                         |
| ------------------------------- | ----------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------- |
| **Config as a domain type**     | `cmd/md-go-validator/main.go` has an unexported `config` struct built by hand | Create `pkg/config.Config` with constructor + validation                                     |
| **Result sum type**             | `types.Result` is one struct with optional `Error`                            | Introduce `ValidResult`, `SkippedResult`, `ErrorResult` with a shared interface              |
| **Break types↔languages cycle** | `types.CodeBlock` imports `languages.Language`                                | Move `Language` (the type) to `pkg/types` or a new `pkg/language` package                    |
| **Filesystem abstraction**      | Direct `os` usage in `pkg/validator.go` and CLI                               | Use `io/fs.FS` or `github.com/spf13/afero` for testability and virtual filesystem support    |
| **Structured errors**           | Plain `fmt.Errorf` wrapping                                                   | Adopt `github.com/larsartmann/go-error-family` for rejection/transient/input classifications |

### 2. Use Well-Established Libraries

| Area                | Current Approach                                 | Established Alternative         | Benefit                                                                 |
| ------------------- | ------------------------------------------------ | ------------------------------- | ----------------------------------------------------------------------- |
| CLI parsing         | Hand-rolled generics parser                      | `cobra` + `viper`               | Config files, env vars, shell completion, sub-commands                  |
| Configuration       | None                                             | `viper` or `koanf`              | `.md-go-validator.yaml`, env overrides, defaults, validation            |
| Filesystem tests    | Real `os` calls                                  | `afero`                         | Fast, deterministic, parallel-safe tests; virtual paths for `--dry-run` |
| Syntax highlighting | Hardcoded ANSI escapes in `pkg/output/output.go` | `chroma` or `lipgloss`          | Maintainable, themeable, correct code rendering in terminal output      |
| Ignore patterns     | None                                             | `github.com/bmatcuk/doublestar` | `.gitignore`-style exclude patterns                                     |
| Progress UI         | None                                             | `cheggaaa/pb` or `bubbletea`    | Progress bars for large directories                                     |

## g) Top #1 Question I Cannot Figure Out Myself

**Should the `Language` type live in `pkg/types` or in a new `pkg/language` package?**

- If it stays in `pkg/languages`, the cycle with `pkg/types` (`CodeBlock` needs `Language`, `languages` needs result types) persists.
- If it moves to `pkg/types`, then `pkg/types` becomes a kitchen-sink package and every validator implementation still has to import it.
- A new `pkg/language` package with _only_ the `Language` branded type and minimal constants would break the cycle cleanly, but it introduces a third package and may feel like overkill.

I can implement any of the three, but the choice has long-term consequences for the modularization proposal already in `docs/modularization/`. Which direction aligns with the intended module boundaries?
